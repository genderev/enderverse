The danger in open source

In Intel’s central processing units, MINIX 3 is the operating system that runs inside the proprietary Intel Management System. The MINIX 3 story poses the question: what if your permissively-licensed open source code is used for something evil?

How does Intel use open source code?

In open source software development, it can be hard to get anyone to use the software you make. When Andrew Tanenbaum developed a self-healing operating system called MINIX, he didn’t immediately see widespread adoption or appreciation for his hard work. He released the third version of the MINIX operating system under the permissive Berkeley license.

When employees from Intel started asking Tanenbaum technical questions about the MINIX operating system, he was happy to engage them. Eventually, Intel secretly modified the MINIX operating system for use in Intel CPUs.

Even though Tanenbaum was grateful for this move, which made MINIX the most widely used operating system in the world, he wished Intel had notified him of their intent to modify MINIX for Intel CPUs.

What’s wrong with Intel using open source code?

On the fifth of May in 2017, Intel released a patch for the Intel CPUs running MINIX 3. Before this security patch, it was possible for hackers to remotely access computers that had an Intel CPU running MINIX 3 without a password.

Intel CPUs run MINIX 3 insidethe Intel Management Engine (ME). Some computers have a Intel Management Engine (ME) which contains an Active Management Technology (AMT) subsystem.

The AMT exists so that system administrators can remotely fix issues with a computer’s network, screen, and keyboard. The AMT allows a remote administrator - or remote hacker - to use a computer as if they were a local user.

The AMT requires remote system administrators to authenticate before beginning a troubleshooting session, but remote access was granted to all who made an attempt at authenticating with the use of any cryptographic hash.

Did Intel fix their security problems?

The software engineers at Intel don’t have the same love for open source that Andrew Tanenbaum, the creator of MINIX 3, does. Even though MINIX 3’s permissive open source license paved the way for the Intel Management Engine, Intel has kept the source code for the Intel Management Engine and related subsystems secret.

Although Intel’s CTO Steve Grobman has denied it, Intel’s Management Engine is a backdoor by definition. Hackers can bypass normal authentication and access a user’s computer by entering their computer through the ME.

To date, the sheer number of exploits affecting the ME - six to date - can be used to make the argument that the ME was designed to be exploited, that is, that the ME is a backdoor.

The Intel Management Engine’s newest exploit - of which there are many - is SA-00086 . Because the Intel Management Engine is more privileged than the operating system, it is actually not possible for a computer affected by this exploit to fix the issue by downloading an operating system update.

How does Andrew Tanenbaum feel?

Even in 2017, the creator of MINIX had regrets working with Intel on ME, saying he would not have worked with them on ME if he knew they were building a “spy engine”.

Andrew Tanenbaum seems to regret his decision working with Intel, but most people who permissively license their code find it rewarding. The question remains: to license permissively or not?
